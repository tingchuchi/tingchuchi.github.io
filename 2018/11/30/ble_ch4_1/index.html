4. 低功耗藍牙規範-鏈結層(1)

發表於 2018-11-30 | 分類於 藍牙 | 閱讀次數:

Link layer 定義了兩個設備如何利用無線電來傳輸訊息。其中包含了封包(Packet)、廣播(Advertising)、資料通道(Data Channel)的詳細定義，也規定了發現(Discover)其他設備的流程、廣播資料(Broadcasting Data)、連接建立、連接管理以及連接中的資料傳輸。2.4GHz ISM 頻段中的干擾、噪音、嚴重衰減(Deep Fade)等無線傳輸系統的挑戰也加深了Link Layer的複雜度。

1 一般描述

1.1 鏈結層狀態(Link Layer States)

Link Layer的操作可以用狀態機來描述，總共有以下五個狀態：

• Standby State (就緒狀態)

• Advertising State (廣播狀態)

• Scanning State (掃描狀態)

• Initiating State (發起狀態)

• Connection State (連接狀態)



圖6. 鏈結層狀態

Link Layer 狀態機只允許一次一個狀態為Active。Link Layer 至少有一個狀態支持 Advertising State或Scanning State。一個Link Layer 的具體實現可以擁有多個獨立的狀態機。

Standby State

BLE裝置上電後，Link layer進入並且維持Standby State直到接到來自Host的命令，在Standby State時，裝置不發送或接收任何的Packet。Standby State 可以進入Advertising State 或是 Initiating State；從其他任意狀態也可以進入Standby State。Standby State 是 Link layer狀態機的中心狀態，儘管是非活動狀態，但也是最為重要。

Advertising State

Advertising State 的裝置負責發送Advertising Packet，也可以發送 Scan Request用來回應主動掃描的裝置。進入Advertising State的裝置稱為Advertiser。Advertising State的裝置停止Advertise後可以進入Standby state。

Discoverable 或是 Connectable 的裝置需要處於Advertising State，想在一定區域內向其他裝置Advertise也需要處於Advertise State。

為了要成為Advertiser，設備必須要有發射器(Transmitter)，當然也可以有接收器(Receiver)，只支援Advertising State的裝置可以設有接收器來降低晶片成本。不過實際上出貨量大的收發器往往會比出貨量小的單一發射器來便宜。

Scanning State

Scanning State 的裝置能夠接收Advertise Channel Packet 。處於Scanning State 的裝置稱為掃描者(Scanner)。Scanning State可由Standby State進入。Scanning State 可分為兩個子狀態： Passive Scanning(被動掃描)及 Active Scanning(主動掃描)。

Passive Scanning(被動掃描)

在Passive Scanning中，裝置只能被動地掃瞄，無法發送任何Packet。因此，Passive Scanning可以在只有Transmitter的裝置中實現。

Active Scanning(主動掃描)

Active Scanning中，Link Layer一旦發現了新的Advertising State 裝置，都會發送Scanning Request(掃描請求)，並等待該Request的回應。Scan Request及Response Packet都會在Advertising channel 中傳輸。

為了使Active Scanning可以更有效率的工作，Scanning response應盡量為Static(靜態資料)，因為就能源消耗而言，Scanning Request及Response這兩個額外的packet會使得data在使用上更加珍貴。然而，原始的Advertising Packet Data則能夠定期變更，因為Advertising Packet總會被接收到。

Initiating State

為了能夠發起連接，Link Layer需要處於Initiating State。Initiating State 可以從Standby State進入。處於Initiating State的裝置稱為Initiator，Initiator其接收器用來監聽自己試圖連接的裝置。如果收到了來自該裝置的Advertise Packet，Link Layer會向其發送Connect Request並進入Connection State。如果initiator不再試圖發起連接，會進入Standby State。

Connection State

從Advertising State 或是Initiating State均可進入Connection State。兩種情況均源自於Initiator向Advertiser發起Connect Request Packet。

Connection State中，兩個設備相互傳輸Data Channel Packets，這也是唯一一個用到Data Channel 的狀態，其他各個狀態均使用到Broadcast Channel。

Connection State有兩個子狀態: Master及Slave。

Master

Master只能從Initiating State進入。為了成為Master裝置，它必須向Peer Device發起連接。Master Device會與 Slave Device 進行通訊並且定義傳輸的時間。Slave Device 只能藉由response packet才能夠傳自己的資料。

Slave

Slave只能從Broadcast State進入。為了成為Slave裝置，它必須向Peer Device進行Broadcast。Slave Device 只能在正確接收來自Master device 的packet才能傳資料。如果Slave想傳更多資料，則必須等待Master device傳送另外的Packet再回復。當然Slave Device可隨時忽略Master Device來達到省電的目的。如此一來，Slave Device可以透過”休眠”來節省大量的能量。

1.2 允許的狀態及角色組合限制 (Permitted State and Role Combination Restrictions)

在一個時刻，狀態機只能處於一種狀態，而Link Layer可同時擁有多個狀態機。這意味著BLE設備在一個狀態機中保持Connection狀態的同時，另一個狀態機可保持Advertising狀態，或者多個Link Layer狀態機同時處於連接狀態，這也是BLE設備實現多個連接的基礎。

以下為同時建立多個連接的場景：

  1. 如果設備A已經跟設備B保持連接，那麼設備A可以執行廣播或掃描操作。
  2. 如果設備A已經跟設備B保持連接，並且設備A是主設備，那麼設備A能夠跟其他從設備C再次建立連接。
  3. 如果設備A已經跟設備B保持連接，並且設備A是從設備，那麼設備A能夠跟其他主設備C再次建立連接。
  4. 如果設備A已經跟設備B保持連接，那麼不能實現設備A掃描，設備B廣播並再次建立連接。設備A與B之間只能維持一個連接狀態機。

1.3 位元順序(Bit Order)

Packet 是1個bit 1個bit傳輸的，但是它們同時也是由資料的bytes組成。當資料的各個 byte傳輸時，是從最低位開始，因此，3bit 的變數 X = 3 會照以下順序發送：

b0b1b2 = 1 1 0

1.4 裝置位址 (Device Address)

裝置使用Device Address(裝置位址)進行辨認。不同的裝置必須具有不同的Device Address

1.4.1 Public Device Address

在通訊系統當中，Device Address是用來唯一辨識一個物理設備的，像是TCP/IP 中的 MAC Address, Bluetooth Classic 的 Bluetooth Device Address。對於Device Address來說一個很重要的特性就是唯一性，否則很可能會造成很多問題。

Public Address屬於一種48-bit的MA-L類型地址，它的結構為：NN:NN:NN:NN:NN:NN。其中前三個byte使用OUI（Organizationally Unique Identifier），後三個byte自由分配。OUI代表了一個指定的組織機構識別碼，全球已經有許多公司申請了自己的OUI。開發者可以從IEEE網站
https://standards.ieee.org/develop/regauth/oui/oui.csv 下載已被分配的MA-L位址。

但是，在BLE當中只有Public Device Adderss是不夠的，有以下原因：

Public Device Address 需要購買，相對於Chip 的成本是不小的開銷
Public Device Address 申請及管理相當繁瑣複雜，再加上BLE裝置數量眾多，維護的成本相當大。
安全考量。BLE很大部分的應用場景為Broadcast Communication，如果知道了Public Device Address，就可以獲取所有的訊息，加深了訊息洩漏的風險。

1.4.2 Random Device Address

為了解決訊息洩漏的風險問題，BLE 規範新增了 Random Device Address，Device Address是在裝置啟動後隨機產生。Random Address分為Static Address和Private Address兩類，Private Address又分為Resolvable private address和Non-resolvable private address兩類。

1.4.2.1 Static Device Address

Static Device Address 是裝置在上電時隨機生成的地址，總共有48 bit，需遵守以下原則：

  1. 最高兩個bit為”11”。
  2. 剩餘的46 bit是一個隨機數，不能全部為0也不可以全部為1
  3. 在一個上電周期內保持不變
  4. 下一次上電的時候可以改變Static Device Address。但不是強制的，因此也可以保持不變。
  5. 如裝置的Static Device Address發生改變，則儲存在Peer devices中的地址將無效，舊地址將無法用於重新連接。

Static Device Address的格式如下圖所示：



圖8. Static Device Address格式

46 bit的隨機數能夠裝置位址唯一性的問題，因為隨機產生重複的機率很小。
同時也可以解決Public Device Address申請所帶來的費用和維護問題。

1.4.2.2 Private Device Address

Static Device Address透過地址隨機生成的方式，解決了部分問題，Private Device Address則更進一步，透過定時更新和地址加密兩種方法來提高藍牙位址的可靠性和安全性。根據地址是否加密，Private Device Address又分為兩類，

Non-resolvable Private Address和Resolvable Private Address。下面我們分別描述。

Non-resolvable Private Address

Non-resolvable private address和Static Device Address類似，不同之處在於，Non-resolvable private address會定時更新。更新的週期則是由GAP規定的，稱作T_GAP(private_addr_int) ，建議值是15分鐘。其格式如下：

  1. 最高兩個bit為“00”。
  2. 剩餘的46bits是一個亂數，不能全部為0，也不能全部為1。
  3. Non-resolvable private address不可以與Public Address一樣
  4. 以T_GAP(private_addr_int)為週期，定時更新。
    Non-resolvable Private Address的格式如下圖所示：


圖9. Non-resolvable Private Address格式
Resolvable Private Address

Resolvable private address不能單獨使用，因此需要使用該類型的位址的話，設備要同時具備Public Device Address或者Static Device Address中的一種。
Resolvable private address透過24-bit亂數和Identity Resolving Key (IRK) 的密碼生成，因此只能被擁有相同IRK的設備掃描到，可以防止被未知設備掃描和追蹤。其格式如下：



圖10. Resolvable Private Address格式

高位24bits是亂數部分稱為prand，其中最高兩個bit為“10”，用於標識網址類型。剩餘的46bits是一個亂數，不能全部為0，也不能全部為1。
低位24bits是亂數和IRK經過hash運算得到的hash值，運算的公式如下：
hash = ah(IRK, prand)。

1.4.2.3 Private Device Address Resolution

當BLE裝置掃描到該類型的藍牙位址後，會使用保存在本機的IRK和該位址中的prand，進行同樣的Hash運算，並將運算結果和位址中的Hash欄位比較，相同的時候，才進行後續的操作。如果一個設備有多個存儲的IRK，設備會重複上述操作，

直到一個IRK的地址解析成功或全部嘗試完成為止。這個過程稱作resolve（解析），這也是Non-resolvable private address/Resolvable private address命名的由來。

Resolvable private address以T_GAP(private_addr_int)為週期，定時更新。所以在廣播、掃描、已連接等過程中，Resolvable private address也可能改變。

1.5 Physical Channel

BLE使用40個Channel 來搜尋裝置、啟動連接以及廣播資料。因為Relaxed Modulation Index(調變率放寬)，BLE的Channel與Bluetooth Classic不同。這意味著在Relaxed Modulation Index之下，每個Channel的Radio Energy(無線電能)會更寬。因此，為了避免鄰近Channel干擾，BLE規範定義Channel寬度為2MHz，而不是Bluetooth Classic的1MHz。

在Link layer中將Physical channel分為幾個部分：

Primary Advertising Channel (廣播通道)

用來初始化Advertising及所有Legacy Advertising Activities，如果Payload在Primary Advertising Channel上傳輸，則為Advertising Payload。

Secondary Advertising channel (次級廣播通道)

位於Channel 0-36。用來處理Extended Advertising Data、Period Advertising Data。

Data Physical Channel (資料實體通道)

位於Channel 0-36。用來處理大部分的資料傳輸。如果Payload在Data Physical channel上傳輸，則為Data payload。

兩個裝置之間希望能用同個共享的Physical Channel來進行通訊。為了要達成這麼目的，裝置的收發器就必須要同時調整到同一個RF channel。但RF channel數量是有限的，在相同的空間與時間當中，可能有許多各自獨立的藍牙裝置。有可能因為其收發器都調整到相同的RF channel，導致Physical Channel 碰撞。

為了減輕這類衝突產生的影響，藍牙在每次傳輸都會先從Access Address開始。Access Address會出現在每一個傳輸Packet的開始處。

Link layer會在一個既定的時間內使用一個Physical Channel。如果裝置之間在Physical Channel中的Timing, Frequency, Access Address達到同步，我們可稱為裝置之間為已連接(Connected)或是已同步(Synchronized)

1.5.1 Advertising Channel及Data Channel

下圖為Physical Channel上與RF Frequency、Channel Index、Channel Type對應的關係：



圖12. Physical Channel 描述


圖13. Physical Channel 分布

由圖可看出，BLE共有3個Advertising Channel以及37個Data Channel。3個Advertising Channel分散在ISM頻段的不同區域，如果他們集中在某個頻段，則可能因為這個頻段的嚴重衰減而造成所有Advertise無法進行。因此，各個Advertising channel 之間(2402MHz,2426MHz,2480MHz)至少相差了24M Hz。Advertising channel的編號為37~39，Data channel 的編號為0~36，用這樣的編號方式來區分使得跳頻演算法的實現變得容易。

Advertising Channel頻點的選擇，原則上是為了遠離Wi-Fi接入點的嚴重干擾。這些接入點通常選擇802.11Channel中的3個，這三個Wi-Fi channel的中心頻率分別為2412MHz,2437MHz,2462MHz，頻寬占了20MHz，意味著這些Wi-Fi channel佔據2402 MHz ~ 2422MHz、2427MHz ~ 2447 MHz、2452 MHz ~ 2472 MHz。



圖14. Physical Channel BT/WIFI 分布

BLE的Advertising Channel 位於2402MHz、2426MHz、2480MHz。也就是說，第1個BLE Advertising Channel低於Wi-Fi channel 1，第2個BLE Advertising Channel介於Wi-Fi Channel 1和Wi-Fi channel 6之間，第3個BLE Advertising Channel高於Wi-Fi Channel 11。3個BLE Advertising Channel皆在這些Wi-Fi Channel的干擾之外。

1.5.1.1跳頻 (Frequency Hopping)

Frequency-hopping Algorithm(跳頻演算法)用在Data Connection 中。跳頻演算法公式如下：

fn+1 = (fn+hop) mod 37

hop 是一個5~16範圍的值，每次跳頻之後，中心頻率會+hop並且mod37。在演算法的實現中，可以先加上hop，如果大於36則減去37。如此一來就不需要除法乘法的複雜運算。



圖15. 跳頻

1.5.1.2 自適應調頻(Adaptive Frequency Hopping )

Adaptive Frequency Hopping能夠將一個已知的Bad Channel 對應到一個已知的 Good Channel，進而減少其他設備對Data packet的干擾。為了要實現這一點，連接中的兩個設備必須要記錄Bad Channel和Good Channel的對應關係。如果跳頻演算法公式所計算出的Channel是Good Channel則使用它；如果是Bad channel則會重新對應到Good Channel的集合。如下圖所示Master需要至少將兩個Data Channel標示為 Good Channel。

Channel 重新對應保證了BLE面對嚴重干擾時依然能告傳輸資料，同時此一機制能讓設備迅速對應對新的干擾。對於Bluetooth Classic，大多數Controller也能在幾秒鐘內應對新的干擾。所以兩者是很容易共存的。



為了輔助重新對應。Host可以告知Controller當前Channel的狀況，告知訊息有可能來自該裝置的其他干擾無線電收發器或是其他地方。大多數藍牙Controller能夠進行被動式頻段掃描，在無需Host任何幫助的情況下，就能確定干擾範圍程度。
